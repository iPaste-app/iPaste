# 快捷动作与自动化方案

## 背景

这个需求描述的是一类可以从 iPaste 主面板里直接触发的本地动作：

- 打开终端。
- 进入指定文件夹。
- 拉取代码。
- 关闭终端。

从产品语义看，它不应该一开始就做成完整的后台工作流系统。当前 iPaste 的核心体验是本地优先、托盘常驻、快速唤出、键盘优先的剪贴板面板，因此更适合先做成“快捷动作”：

- 用户提前创建动作。
- 主面板里可以搜索、选中、双击或回车运行。
- 动作在本机执行。
- 执行结果和日志在应用内可见。

后续如果增加条件分支、定时、变量、AI 生成步骤，再把能力扩展成“自动化”或“工作流”。

## 命名建议

推荐对用户使用“快捷动作”，对内部能力使用“自动化”。

可选命名：

| 名称 | 适合度 | 说明 |
| --- | --- | --- |
| 快捷动作 | 高 | 贴合主面板手动触发，轻量、清晰。 |
| 自动化 | 中高 | 能覆盖多步骤执行，但用户可能期待后台、定时、条件触发。 |
| 工作流 | 中 | 更像复杂流程编排，第一版容易显得过重。 |
| 智能操作 | 中低 | 适合 AI 参与后的命名，当前阶段不够准确。 |
| 快捷指令 | 中 | 用户容易理解，但和系统级 Shortcuts / 快捷指令有语义重叠。 |

建议第一版模块名：

- 顶部或分类入口：`动作`
- 新建按钮：`新建快捷动作`
- 执行按钮：`运行`
- 内部类型：`automation`

## 当前项目分析

### 已具备的基础

iPaste 当前已经具备实现这个能力的主要基础：

- Tauri 2 + Vue 3 桌面应用结构已经成型。
- 主面板已经支持搜索、分类、卡片列表、选中态和右侧详情。
- 面板已经支持键盘导航，`Enter` 会执行当前选中项。
- 卡片已经支持双击执行。
- Rust 侧已有 SQLite 存储层和 Tauri command 暴露模式。
- 应用已经有托盘、全局快捷键、窗口显示隐藏和 macOS 非激活面板处理。

相关位置：

- `src/App.vue`：主面板、键盘导航、右键菜单、双击/回车入口。
- `src/stores/ipasteStore.ts`：Pinia 状态、选中项、应用选中项。
- `src/types.ts`：前端数据模型。
- `src/lib/ipasteApi.ts`：前端 Tauri command 封装。
- `src-tauri/src/lib.rs`：SQLite、Tauri commands、窗口、剪贴板、粘贴自动化。
- `src-tauri/capabilities/default.json`：Tauri 权限能力声明。

### 当前缺口

当前条目模型只表达剪贴板内容：

- `ClipType` 包含 `text`、`link`、`color`、`image`、`file`、`html`。
- `ClipViewItem` 只区分历史记录和分类条目。
- `applySelected` 现在的语义是“把剪贴板内容写入系统剪贴板并粘贴”。

因此需要新增一类“面板条目”：

- 剪贴板条目：执行语义是粘贴。
- 快捷动作条目：执行语义是运行动作。

另外，当前 Tauri capability 里只允许 `process:allow-restart`，用于应用重启，不具备用前端直接执行 shell 命令的权限。Rust 侧虽然使用过 `std::process::Command` 打开 macOS 辅助功能设置，但没有通用命令执行器。

结论：执行本地命令可行，但应该通过 Rust 新增受控 Tauri command 实现，不应该让前端直接拥有任意 shell 执行能力。

## 可行性结论

### 高可行

适合第一版实现：

- 手动触发快捷动作。
- 后台运行本地命令。
- 支持指定工作目录。
- 捕获 stdout、stderr、退出码、开始时间、结束时间。
- 在主面板右侧展示最近一次运行状态和日志。
- 支持双击和回车运行。

示例：

```text
名称：拉取 iPaste 代码
工作目录：E:\code\idea\ipaste-new
命令：git pull
运行方式：后台运行
```

这个例子中，“进入指定文件夹”不需要作为独立步骤，可以映射为命令执行的 `cwd`。如果选择后台运行，也不需要真的打开和关闭终端。

### 中等可行

可以在第二阶段做：

- 在系统终端中运行。
- 命令运行前弹出确认。
- 运行中取消。
- 多步骤串行执行。
- 动作分类、排序、置顶。
- 动作运行历史。

系统终端模式跨平台差异较大：

- Windows 可能涉及 Windows Terminal、PowerShell、cmd。
- macOS 可能涉及 Terminal、iTerm、osascript。
- 不同终端的“执行完自动关闭”行为不一致。

因此建议先把“后台运行 + 应用内日志”做扎实，再扩展真实终端。

### 暂不建议第一版做

- 定时自动运行。
- 后台条件触发。
- 云同步动作。
- AI 自动生成并立即执行动作。
- 复杂分支、循环、变量表达式。
- 无限制任意 shell 脚本执行。

这些能力会明显扩大安全风险和交互复杂度。

## 产品交互方案

### 主面板入口

建议在分类栏中增加一个固定入口：

- `历史`
- 用户分类
- `动作`

`动作` 不是普通剪贴板分类，不参与当前分类同步逻辑。

进入 `动作` 后：

- 搜索框搜索动作名称、命令、工作目录。
- 列表展示动作卡片。
- 回车运行选中动作。
- 双击运行动作。
- 右键菜单提供运行、编辑、复制、删除、查看日志。

### 动作卡片

动作卡片建议展示：

- 标题：动作名称。
- 副标题：命令摘要和工作目录。
- 图标：终端、播放、代码、文件夹等。
- 状态：就绪、运行中、成功、失败。
- 最近运行时间。
- 失败时用轻量错误状态提示，不在卡片里塞大量日志。

示例：

```text
拉取 iPaste 代码
git pull · E:\code\idea\ipaste-new
上次成功 · 2 分钟前
```

### 右侧详情

选中动作时，右侧详情面板展示：

- 动作名称。
- 工作目录。
- 命令。
- 运行方式。
- 最近一次状态。
- 退出码。
- 耗时。
- stdout / stderr 日志。
- 运行按钮。
- 停止按钮，运行中可见。

右侧详情比弹窗更适合当前 iPaste 的密集工具界面。

### 创建和编辑

第一版可以使用独立编辑弹窗，字段保持克制：

- 名称。
- 工作目录。
- 命令。
- 运行前确认。
- 运行方式：后台运行。

暂不需要可视化流程编排器。对当前需求来说，一个命令加工作目录已经能覆盖大量场景。

### 执行反馈

运行时建议：

- 主面板保持打开。
- 动作卡片显示运行中状态。
- 右侧日志实时追加。
- 成功后显示成功状态。
- 失败后保留错误日志。

如果用户从主面板触发后希望自动关闭面板，可以作为动作配置：

- 运行后保持面板打开。
- 运行成功后关闭面板。

默认建议保持面板打开，因为命令执行需要反馈。

## 数据模型建议

### 前端类型

建议新增：

```ts
export type AutomationRunMode = "background" | "terminal";

export type AutomationStatus = "idle" | "running" | "success" | "failed" | "cancelled";

export type AutomationAction = {
  id: string;
  name: string;
  command: string;
  cwd?: string | null;
  runMode: AutomationRunMode;
  confirmBeforeRun: boolean;
  closePanelOnSuccess: boolean;
  sortOrder: number;
  createdAt: string;
  updatedAt: string;
  lastRun?: AutomationRunSummary | null;
};

export type AutomationRunSummary = {
  id: string;
  automationId: string;
  status: AutomationStatus;
  exitCode?: number | null;
  startedAt: string;
  finishedAt?: string | null;
  durationMs?: number | null;
};
```

面板条目可以抽象为：

```ts
export type PanelItem =
  | { kind: "clip"; item: ClipViewItem }
  | { kind: "automation"; item: AutomationAction };
```

这样 `applySelected` 可以根据 `kind` 分流：

- `clip`：调用现有 `applyClip`。
- `automation`：调用新增 `runAutomation`。

### SQLite 表

建议新增表：

```sql
CREATE TABLE IF NOT EXISTS automations (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  command TEXT NOT NULL,
  cwd TEXT,
  run_mode TEXT NOT NULL DEFAULT 'background',
  confirm_before_run INTEGER NOT NULL DEFAULT 0,
  close_panel_on_success INTEGER NOT NULL DEFAULT 0,
  sort_order INTEGER NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS automation_runs (
  id TEXT PRIMARY KEY,
  automation_id TEXT NOT NULL,
  status TEXT NOT NULL,
  exit_code INTEGER,
  stdout TEXT NOT NULL DEFAULT '',
  stderr TEXT NOT NULL DEFAULT '',
  started_at TEXT NOT NULL,
  finished_at TEXT,
  duration_ms INTEGER,
  FOREIGN KEY(automation_id) REFERENCES automations(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_automations_sort_order
  ON automations(sort_order, created_at);

CREATE INDEX IF NOT EXISTS idx_automation_runs_automation_started
  ON automation_runs(automation_id, started_at DESC);
```

第一版不建议把动作塞进 `category_items`，原因：

- 剪贴板条目和动作的执行语义不同。
- 动作不应该参与剪贴板云同步。
- 动作需要运行状态和日志。
- 未来动作可能有步骤、变量、权限确认等专属字段。

## Tauri Command 设计

建议新增前端 API：

```ts
listAutomations(): Promise<AutomationAction[]>
createAutomation(input): Promise<AutomationAction>
updateAutomation(id, input): Promise<AutomationAction>
deleteAutomation(id): Promise<void>
runAutomation(id): Promise<AutomationRunSummary>
getAutomationRun(id): Promise<AutomationRunDetail>
stopAutomation(runId): Promise<void>
```

Rust command 建议：

```rust
#[tauri::command]
fn list_automations(state: tauri::State<'_, AppState>) -> Result<Vec<AutomationAction>, String>

#[tauri::command]
fn create_automation(state: tauri::State<'_, AppState>, input: AutomationInput) -> Result<AutomationAction, String>

#[tauri::command]
fn update_automation(state: tauri::State<'_, AppState>, id: String, input: AutomationInput) -> Result<AutomationAction, String>

#[tauri::command]
fn delete_automation(state: tauri::State<'_, AppState>, id: String) -> Result<(), String>

#[tauri::command]
fn run_automation(app: tauri::AppHandle, state: tauri::State<'_, AppState>, id: String) -> Result<AutomationRunSummary, String>
```

运行事件建议：

```text
ipaste://automation-run-started
ipaste://automation-run-output
ipaste://automation-run-finished
```

事件载荷：

- `runId`
- `automationId`
- `stream`: `stdout` 或 `stderr`
- `chunk`
- `status`
- `exitCode`
- `startedAt`
- `finishedAt`

这样前端可以实时更新卡片和详情日志。

## 命令执行器设计

第一版建议只支持后台运行。

### Windows

可用 PowerShell 执行：

```text
powershell.exe -NoProfile -ExecutionPolicy Bypass -Command "<user command>"
```

并设置：

- `current_dir(cwd)`
- stdout piped
- stderr piped

也可以根据配置选择 `cmd.exe /C`。但第一版建议固定 PowerShell，减少配置复杂度。

### macOS

可用 shell 执行：

```text
/bin/zsh -lc "<user command>"
```

并设置：

- `current_dir(cwd)`
- stdout piped
- stderr piped

### 安全边界

需要明确：

- 只允许用户手动创建的命令运行。
- 第一次运行危险命令时建议确认。
- 不自动从剪贴板、云端、网页生成命令并执行。
- 动作默认不云同步。
- 工作目录必须存在。
- 命令不能为空。
- 命令长度限制，例如 4000 字符。
- 日志长度限制，例如单次最多保留 200 KB stdout 和 200 KB stderr。
- 单个动作同时只允许一个运行实例，避免重复触发。
- 支持超时，例如默认 10 分钟，可后续做成配置。

### 关于打开终端和关闭终端

用户描述的“打开终端，进入目录，拉取代码，关闭终端”可以拆成两种实现：

第一版推荐：

```text
后台运行：git pull
cwd：指定文件夹
日志：显示在 iPaste 右侧详情
```

后续增强：

```text
系统终端运行：打开终端窗口并执行 cd + git pull
完成后根据配置关闭或保留终端
```

不建议第一版直接做终端自动开关，因为跨平台终端控制不可预测，且失败反馈容易丢失。

## Store 和面板改造

### Store

`ipasteStore` 建议新增：

- `automations`
- `automationRuns`
- `selectedMode` 或把 `selectedCategoryId` 扩展支持 `actions`
- `visiblePanelItems`
- `selectedPanelItem`
- `runAutomation`

原有：

```ts
const selectedItem = computed(() => visibleItems.value[selectedIndex.value]);
```

可以演进为：

```ts
const selectedPanelItem = computed(() => visiblePanelItems.value[selectedIndex.value]);
```

执行入口：

```ts
async function applySelected() {
  const selected = selectedPanelItem.value;
  if (!selected) return;

  if (selected.kind === "clip") {
    await applyClipItem(selected.item);
    return;
  }

  await runAutomation(selected.item);
}
```

### App.vue

`handlePanelKey` 里的 `Enter` 不需要改变用户行为，只改变内部执行分流。

双击也一样：

- 剪贴板卡片：粘贴。
- 动作卡片：运行。

右键菜单需要根据条目类型展示不同菜单：

剪贴板条目：

- 粘贴
- 复制
- 重命名
- 移动到
- 删除

动作条目：

- 运行
- 编辑
- 复制动作
- 查看日志
- 删除

### DetailPane

`DetailPane` 当前只支持 `ClipViewItem`。建议改造成：

- `ClipDetailPane`
- `AutomationDetailPane`
- 外层 `DetailPane` 根据 `PanelItem.kind` 分发。

这样能避免把动作日志和剪贴板预览混在一个组件里。

## MVP 范围

第一版建议范围：

1. 新增 `动作` 固定入口。
2. 新建、编辑、删除快捷动作。
3. 动作字段：名称、工作目录、命令、运行前确认。
4. 后台运行命令。
5. 捕获 stdout、stderr、退出码。
6. 卡片展示运行状态。
7. 右侧详情展示最近一次日志。
8. 双击和回车运行。

暂不做：

- 定时。
- 条件触发。
- 多步骤可视化编排。
- 云同步。
- AI 生成动作。
- 系统终端运行。
- 复杂变量。

## 风险与注意事项

### 安全风险

本地命令执行天然危险。需要避免：

- 从云端同步命令后自动执行。
- 从剪贴板内容生成命令后自动执行。
- 无确认执行删除、格式化、强制推送等危险命令。
- 日志泄漏密钥。

第一版建议动作仅保存在本机，不纳入现有云同步。

### 跨平台风险

后台运行相对稳定。系统终端运行风险更高：

- 终端路径不固定。
- 执行完关闭行为不同。
- shell quoting 容易出问题。
- 中文路径、空格路径需要额外测试。

### UI 风险

不要把动作塞进现有剪贴板卡片里硬套。动作需要：

- 不同图标。
- 不同执行文案。
- 不同详情面板。
- 不同右键菜单。

否则用户会混淆“粘贴”和“运行命令”。

### 性能风险

命令日志可能很大。需要：

- 限制单次日志大小。
- 流式追加时节流更新 UI。
- 只加载最近一次或最近几次运行记录。

## 推荐结论

这个需求值得做，且适合从“快捷动作”开始做。

第一版最推荐的产品定义：

> 在 iPaste 主面板中创建和运行本地快捷动作。动作可以指定工作目录和命令，用户通过搜索、双击或回车手动触发，iPaste 在后台执行并展示运行日志。

这个定义既符合当前产品的快速面板体验，也能覆盖“进入目录并拉取代码”这样的真实使用场景，同时避免一开始进入复杂工作流系统带来的安全和交互负担。
